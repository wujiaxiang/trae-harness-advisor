# 任务编排与文档质量最佳实践（面向弱执行 LLM）

> **单一真源**：本文件是 Planner / Generator / Evaluator / Contract 编写的共享底层方法论。各角色 Skill 只保留最关键的内联硬规则，细节回到本文。
> **核心假设**：执行任务的 LLM（云端 SubAgent，常为国产免费模型）**没有编排者聪明、没有项目背景、只能严格按文档执行**。你写的任务文档（Stage 验收要点 / contract.md / spec / tasks）的质量，直接决定执行结果的质量。
> **来源**：沉淀自真实国产 LLM 驱动开发实践（DeepSeek/Qwen 等执行端），v4.6 内化进 harness。

---

## 一、三条核心原则（所有角色必须内化）

### 原则 A：一个验证命令 = 一个子任务的完成边界
如果一个子任务/验收要点**不能用「一个可运行命令 + 一个明确的 ✅/❌ 输出」判断是否完成**，它就还需要继续拆，或需要先写一个验证脚本。
- ❌ 「实现登录功能」——无法一命令判定
- ✅ 「`pytest tests/unit/test_login.py -v` 输出 `0 failed`」

### 原则 B：「不需要做什么」比「需要做什么」更重要
LLM 面对模糊任务会做它认为"顺手"的事。**不显式排除的范围，就是它自由发挥的空间。** 每个 Stage/子任务都必须显式写「不包含 / 不要改」。
- ❌ 只写「验证 Coinglass API 连通性」
- ✅ 追加「不包含其他 8 个数据源；不改 feature_builder.py；不启动 Analyst 服务」

### 原则 C：停止条件必须写明，LLM 不会自己决定停
LLM 为了"完成任务"会持续尝试，包括做出错误决策。**必须明确写：什么情况停下来、输出什么、等谁决定。**（见第四节 7 种必停情况）

## 二、任务拆分：先判阶段类型，再套策略（Planner 主用）

### 2.1 三种阶段类型

| 阶段 | 典型特征 | 主要产出 | 拆分原则 |
|------|---------|---------|---------|
| **开发 Dev** | 有具体文件要改；可用单测验证 | 改后代码 + 测试全绿 | 按文件分组（每 1–2 文件一个子任务），测试命令=验证命令 |
| **联调 Integration** | 调外部 API；有环境变量依赖；报错来自网络/权限/格式 | 外部系统连通 + 关键字段有真实数据 | 每个外部系统一个子任务；API Key 准备为独立步骤；每子任务带失败速查表 |
| **验收 Acceptance** | 面向业务结果；需时间积累；人工最终确认 | 业务指标达标的证明 | 把业务标准翻译成**可查询指标**；写"你看到什么就算通过"（不是"测试通过"） |

> 映射到 harness：阶段类型影响 Stage 的 `contract_mode`（Dev 早期常 codraft、联调/验收常 planned）与 `pattern`（可拆的联调=fanout、需判类=classify）。

### 2.2 上下文预算：决定一个 Stage/对话装多少
```
对话容量 ≈ 任务文档行数 + 需读代码文件行数 + 命令输出行数
上限建议 ≈ 3000 行
```
逐项估算，超上限就**拆对话/拆 Stage**。例：验证 9 个数据源（9 命令×20 行输出 + 修复代码）→ 拆成「1 基线扫描 + 9 独立 + 1 汇总」。

### 2.3 串行 vs 并行（直接决定 depends_on 与 fanout）
- **串行（A→B）**：B 需要 A 的输出产物；或 A 失败则 B 无意义。→ 标 `depends_on`。
- **并行（A∥B）**：A、B 针对不同外部系统；A 失败不影响 B。→ 无依赖，可并发，天然适合 `pattern=fanout`。

### 2.4 拆完自检 4 问
1. 能用一个命令验证是否完成吗？（否→继续拆）
2. 范围边界清楚吗？写了"不需要做什么"吗？（否→补一行"不需要…"）
3. 失败时下一步是什么？文档有指引吗？（否→补速查表/诊断步骤）
4. 人工在什么时候介入？写明停止条件了吗？（否→补"遇到以下情况停下等人工"）

## 三、子任务/Contract 的 6 段式结构（Contract 编写、Orchestrator 主用）

每个子任务（或 contract.md 的验收项）应尽量含以下 6 段，**不省略**：

1. **前置检查命令**：服务是否在线 / 环境变量是否已设（带 ✅/❌）。
2. **执行命令**：完整可运行（无 `...` 省略）；python 含 `sys.path.insert(0,'.')`；**内嵌 ✅/❌ 判断，不能只打印原始数据**；环境变量写命令头 `KEY=val python3 -c "..."`。
3. **期望输出**：具体字段 + 合理值范围（不写"类似这样"）。
4. **失败速查表**：`| 看到的报错 | 最可能原因 | 做这个（不要做这个） |`——必须具体到操作 + **禁止事项**（如"不要改 Header 名"）。
5. **修改位置**（如需改码）：文件→函数级；显式写"只改这个，不要改其他"。
6. **通过标准**：全部满足才算完成，每条含可观测数值/状态。

> 常见错误：①验证命令只 try/except 判"不报错"（应检查关键字段有合法值）；②速查表只写"检查一下"（应到操作步骤+禁止项）；③范围边界没显式写；④停止条件用模糊词"遇到问题就停"。

## 四、执行时自我监控（Generator 主用）

### 4.1 执行前 5 问（任一"不知道"就先找编排者/人工补文档）
- 这次只做哪几件事？（>3 件提醒拆开）
- 哪些文件可读/不需读？
- 每步完成标准是什么？（须有 ✅/❌）
- 遇到什么要停下等人工？
- 完成后输出什么格式的状态报告？

### 4.2 七种必须立即停（输出 `[BLOCKED: 原因]` 等人工，不自由发挥）
1. **HTTP 401/403**——Key/权限问题，不猜 Key 格式、不改认证代码。
2. **同一问题改 >3 次仍失败**——诊断方向本身错了，列出 3 次尝试+原始报错，等人工给新方向。
3. **需要改 >2–3 个文件**（级联修改）——超预期范围，列实际涉及文件，等确认。
4. **全量测试新增失败**（回归）——列新增失败项，无权自行忽略。
5. **涉及真实资金/生产凭证/生产环境**——硬规则，无授权不执行。
6. **代码里方法/字段不存在**——这是文档 bug 不是代码 bug，grep 实际存在的，等人工确认。
7. **需要人工提供的信息**（API Key/账号/决策）——不能从代码文档自动获取的，停下索取。

### 4.3 按需读文件，不预读
运行验证命令 → 看报错 → 报错指向哪个文件 → 才读那个文件的 ±10~20 行（`grep -n` 定位）。不要一开始 `cat` 一堆"可能用到"的文件。

### 4.4 报错诊断标准步骤（不猜，按顺序）
1. 打印完整报错（`2>&1 | cat`，不截断 traceback）。
2. 判类型：HTTPError→网络/认证查速查表；ImportError→装依赖；AttributeError→方法名错，grep 实际码；KeyError→JSON 字段变了，打印实际 keys；ValueError/TypeError→打印实际值。
3. 定位报错行（traceback 末行文件+行号）。
4. 只读那处 ±10 行。
5. 改 → 重跑验证命令 → 看结果。
6. 同方向 3 次仍失败 → `[BLOCKED]`。

## 五、状态报告格式（Generator 主用，固定不省略）

**每个子任务后**：
```
[T{阶段}.{序号} ✅ DONE] 完成内容一句话；关键结果=具体数值/状态；下一步=T{...}
[T{阶段}.{序号} ❌ FAILED] 失败原因；报错前100字；已尝试次数/方向；需要人工提供什么
```

**整个 Stage 完成后**（下一 Stage 的起点）：
```markdown
### 通过的验收项
- ✅ [项]：[实际结果含数值]
### 可接受的未通过项（不阻断）
- ⚠️ [项]：[原因，如"付费API，标 missing，属设计预期"]
### 下一步需要人工提供
- [ ] [具体信息+获取途径]
### 给下一次对话的关键信息
- [不在文档里但下一步需知道的：实际字段列表、确认的 model ID 等]
```

> 自检快问：**"把当前工作截图发给项目负责人，他能看懂我在做什么、做到哪一步吗？"** 不能→状态报告不够清晰或已跑偏。

## 六、验收判定（Evaluator 主用）
- **"你看到什么就算通过"**：验收标准写可观测现象（"日志里没有 CRITICAL"），不写"测试通过/返回200"。
- **一命令边界**：每个验收项对应一个可运行查询命令，30 秒内出结果。
- **不放水**：不确定往低打分；可接受的未通过项要显式标 ⚠️ 并写清"为何不阻断"。
